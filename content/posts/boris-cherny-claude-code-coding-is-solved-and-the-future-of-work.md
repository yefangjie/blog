# Boris Cherny 和 Claude Code：当他说 coding is solved，他真正改写的是软件生产与工作的分工

最近围绕 Boris Cherny 和 Claude Code，我连续看了两场访谈：一场是 **Sequoia**，一场是 **CNBC**。

如果只看标题，很容易把它们理解成两类老生常谈：

- 一类是“AI 写代码越来越强了”
- 另一类是“AI 会改变工作方式”

但把两场放在一起看，真正值得记住的不是其中任何一句口号，而是一个更大的判断：

**真正正在变化的，不只是 coding，而是人类如何参与软件生产，以及知识工作如何围绕 AI 系统重新分工。**

Boris 说 **coding is solved**，CNBC 那场又把话题往 **future of work and productivity** 拉了一步。把这两句拼起来，你会发现重点根本不是“程序员会不会消失”，而是：

> **软件生产的最小单位在变，工作中的责任边界也在变。**

## 一、当 Boris 说 coding is solved，他真正想表达的不是“代码不重要了”

很多人听到这句，第一反应是要么兴奋过头，要么防御过头。

但如果认真拆，他的意思更接近于：**对于某些高频、强上下文、可迭代的软件工作，手写代码已经不再是主要瓶颈。**

这和“AI 能不能补全函数”“能不能写一个 demo”完全不是一个层次的问题。

真正大的变化在于，代码正在慢慢从核心动作退成中间层：

- 目标先由人定义
- 实现先由系统推进
- 人在关键节点做判断、约束和验收

所以，未来值钱的未必是最会手写代码的人，而是最会把目标翻译成系统可执行工作流的人。

## 二、Claude Code 最重要的变化，不是更强的 IDE，而是软件生产单位变了

Sequoia 那场里最有启发的，不是某个具体功能，而是 Boris 对 **loops** 的强调。

这句话非常关键，因为它把焦点从“单次生成能力”移到了“持续推进能力”。

过去的软件生产基本是这样：

- 人理解需求
- 人拆任务
- 人写代码
- 人调试
- 人提交 PR
- 人 review

而 Claude Code 所代表的新范式更像这样：

- 人定义目标
- Agent 推进实现
- loops 负责重试、补全、修正
- 人负责最终判断和边界控制

也就是说，真正被改写的不是某一段代码怎么写，而是：

**软件生产开始从一次性执行，变成围绕目标持续循环。**

这才是 “coding is solved” 背后真正有分量的地方。

## 三、CNBC 那场补上的，是“未来工作”这层：不只是工程师，知识工作都会被重排

如果 Sequoia 那场更像在讲软件生产方式，CNBC 那场更像是在问：

**当 AI 开始承担越来越多中间执行工作，人类在工作中的位置会往哪里移动？**

这就把话题从 AI coding 扩展到了 future of work。

因为一旦你接受一个前提——系统可以替人执行越来越多具体动作——那后面被改写的就不只是工程师，而是大量知识工作者：

- 产品经理更像目标定义者
- 运营更像流程编排者
- 分析师更像判断与质检者
- 工程师更像系统监督者与异常处理者

这也是为什么我觉得这两场访谈应该合在一起看。

单看其中一场，你可能只会觉得“Claude Code 很猛”。

但放在一起，你会意识到：

> **AI 真正改变的，不只是产能，而是工作中的分工方式。**

## 四、未来最重要的竞争力，正在从“亲手做”转向“设计系统让结果稳定出现”

不管是写代码，还是更广义的知识工作，接下来最重要的问题可能都不是：

- 你会不会亲手做
- 你做得快不快

而是：

- 你能不能把目标定义清楚
- 你能不能让系统稳定推进
- 你能不能发现异常、纠偏、收口
- 你能不能把可靠性和速度同时守住

所以这场变化并不只是效率工具升级，而更像一次组织逻辑的迁移：

**从“人直接完成任务”，转向“人驱动系统完成任务”。**

如果这个判断成立，那未来最值钱的人不会只是“技能最熟练的人”，而会是：

**最会设计流程、管理 loops、控制风险、把 AI 系统压进真实工作里的人。**

>
> Claude Code 代表的变化，不只是 AI 写代码更快，而是软件生产开始从一次性执行转向持续循环。
>
> AI 对工作的真正影响，不只是替代某些动作，而是重排人与系统之间的分工边界。

## 常见问题（FAQ）

### Boris 说 coding is solved，到底是什么意思？
更准确的理解不是“代码消失了”，而是很多软件工作正在从“人直接写代码”转向“人驱动系统产出代码”。

### Claude Code 真正改变了什么？
它改变的不只是 IDE 体验，而是软件生产的最小工作单元：从手写代码，转向定义目标、跑 loops、验结果、控边界。

### 这和 future of work 有什么关系？
因为同样的逻辑会外溢到更多知识工作：人不再负责所有中间执行动作，而更像目标设定者、监督者和异常处理者。

### 一句话结论是什么？
AI 真正改写的，不只是 coding，而是“人如何参与工作”的旧分工。
