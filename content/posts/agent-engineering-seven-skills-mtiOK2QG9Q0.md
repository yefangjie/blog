上周我看到一个招聘启事，差点笑出声。

它写着：招聘一名 **Prompt Engineer**，要求具备 **分布式系统、API 设计、机器学习运维、安全工程和产品管理** 经验。

说实话，这根本不是在招提示词工程师，这像是在招 **五个人**。

但仔细想想，这个 JD 本身也不算错。错的不是要求太多，而是这个岗位名字起错了。

因为当你真正开始构建一个能在现实世界里执行任务、调用工具、访问数据、做出决策的 AI Agent 时，你会很快发现：

> **问题早就不是“提示词怎么写得更漂亮”，而是“系统怎么设计得足够可靠”。**

这也是这支视频最核心的判断：

> **Prompt Engineering 只是起点，Agent Engineering 才是把 AI 做进生产环境的真本事。**

作者把这件事拆成了 7 个关键能力。我看完以后觉得，这个框架非常适合拿来给所有想做 AI Agent 的人校正预期。

---

## 一、Prompt 工程像菜谱，Agent 工程像当主厨

视频里用了一个很妙的比喻。

一个厨师不是只会照着菜谱做饭。任何人都可以照步骤把菜做出来，但真正的厨师理解的是：

- 食材
- 火候
- 节奏
- 厨房协作
- 食品安全
- 出错时怎么即兴补救

菜谱只是起点。

放在 AI 世界里也一样：

- **Prompt Engineering** 像菜谱
- **Agent Engineering** 像当主厨

如果你现在还停留在“我会写几个好 prompt”，那你更接近会照方抓药；如果你想做的是能落地、能执行、能抗故障的系统，那你必须开始学主厨那一套。

---

## 二、真正能把 Agent 做进生产环境的人，至少要补齐这 7 个能力

### 1. 系统设计（System Design）

作者认为，第一能力不是 prompt，而是 **系统设计**。

因为你在做的已经不是“一个会聊天的模型”，而是一整个协作系统：

- LLM 在做决策
- Tools 在执行动作
- 数据库在存状态
- 可能有多个模型或子 Agent 分工
- 不同模块之间还要协调、回滚、重试

这本质上就是架构设计问题：

- 数据怎么流转？
- 某个组件挂了怎么办？
- 多个模块之间如何协同而不互相踩踏？
- 一个任务需要三个 specialist 时怎么编排？

这也是为什么很多真正把 Agent 做起来的人，最后看上去越来越像后端架构师，而不是“提示词魔法师”。

**我的看法：Agent 不是魔法，它只是把传统软件工程里本来就复杂的系统设计问题，和语言模型的不确定性绑在一起了。**

---

### 2. 工具与契约设计（Tool / Contract Design）

Agent 与现实世界交互，靠的是工具。

而每一个工具，本质上都在对 Agent 说：

- 你给我什么输入
- 我会返回什么输出
- 哪些字段是必须的
- 哪些值是合法的

如果这个“契约”定义得含糊，LLM 就会自己脑补。

问题在于，LLM 的想象力在写文案时可能是优点，在处理退款、查用户、发起交易时就很容易变成灾难。

举个典型例子：

如果一个查用户信息的工具 schema 只写着：

- `user_id: string`

那模型可能会传：

- `John`
- `user123`
- 甚至一句自然语言

但如果你把契约写得更严格：

- 必须匹配什么格式
- 给出合法示例
- 哪些字段 required

那模型的可控性就会明显提高。

**这一步很多人会忽略，但它其实是“Agent 像不像个靠谱系统”的分水岭。**

---

### 3. 检索工程（Retrieval Engineering）

大多数生产级 Agent 都离不开 RAG，也就是 **Retrieval Augmented Generation**。

简单说，就是不指望模型记住一切，而是在回答或执行前，先检索相关资料，再把相关内容送进上下文。

很多人会以为：

- 把文档切一切
- 做个 embedding
- 相似度搜一下

不就完了吗？

其实远没那么简单。

检索质量，几乎直接决定 Agent 性能的天花板。

这里至少有几个关键问题：

- 文档怎么切块？
  - 切太大，关键信息被稀释
  - 切太小，语义上下文断裂
- embedding 模型是否真的把相关概念放在一起？
- 是否要做 re-ranking，把真正最相关的结果排到最前？
- 如何避免把错误、噪音或过时信息喂给模型？

如果喂进去的上下文本身就是垃圾，模型只会很自信地拿着垃圾继续输出。

**我很认同作者这一点：很多 Agent 表现差，不是模型不够强，而是喂进去的检索结果太烂。**

---

### 4. 可靠性工程（Reliability Engineering）

这部分很关键，也最容易被演示视频故意跳过。

因为 Demo 只需要成功一次，生产环境要求的是：

> **99 次里也不能 20 次翻车。**

Agent 系统里，每走一步都可能失败：

- LLM API 超时
- 工具返回异常
- 第三方接口限流
- 上下文损坏
- 重试逻辑把系统拖进循环
- 中间状态没存好，任务半截死掉

所以你需要像做生产系统那样设计 Agent：

- 超时控制
- 重试机制
- 幂等性
- fallback 路径
- 日志与追踪
- 状态持久化
- 故障隔离

说白了：

**能跑起来不难，能稳定跑才难。**

如果你的 Agent 每次都像赌博，那它不叫产品，只能叫演示道具。

---

### 5. 评估与迭代（Evaluation）

很多人做 Agent 最大的问题是：

- 改了 prompt
- 改了流程
- 换了模型
- 感觉“这次好像更好了”

但这个“好像”在工程里基本等于没说。

你必须建立自己的评估体系：

- 哪些任务算成功？
- 错在哪类环节最多？
- 是检索问题，还是工具调用问题？
- 修改之后是否真的提升了成功率？

没有评估，所有优化都容易变成玄学。

这也是为什么很多 Agent 项目会陷入一种错觉：

- demo 越看越聪明
- 系统却没变得更可靠

**没有 evaluation 的 Agent 迭代，本质上就是在靠直觉调一台复杂机器。**

---

### 6. 安全与权限设计（Security）

一旦 Agent 不只是“回答问题”，而是开始：

- 调数据库
- 发邮件
- 下单退款
- 修改系统状态
- 访问公司内部知识库

安全问题就立刻从“重要”升级成“致命”。

这里最关键的不是口头上说“要注意安全”，而是：

- 权限边界怎么划？
- 哪些动作必须人工确认？
- 工具能访问哪些数据？
- 如何防 prompt injection？
- 如何避免越权调用？
- 如何记录审计日志？

很多人以为安全是最后补的东西，但在 Agent 系统里，**安全边界本身就是系统设计的一部分**。

否则模型一旦接上工具，它犯的就不是“回答错了”，而是“真的把错误执行了”。

---

### 7. 产品与工作流思维（Product Thinking）

最后一个能力，很多纯技术人会忽略，但作者提得很对：

> 你不是在做一个会说话的模型，而是在设计一个完成任务的工作流。

这意味着你需要思考：

- 用户真正想完成什么任务？
- 哪些步骤应该交给 Agent？
- 哪些步骤应该保留给人？
- 失败时用户怎么接管？
- 什么样的交互方式才不让人迷失？

做不好这一层，就算模型和工具再强，也很容易沦为“技术上能跑，产品上没人想用”。

因为用户要的不是一个炫技 demo，而是一个：

- 能帮他省时间
- 能替他少犯错
- 能在关键时刻让人接管的系统

## 三、为什么“Prompt Engineer”这个说法越来越不够用了？

这支视频最打动我的地方，不是它列了 7 个能力，而是它非常准确地点破了一个现实：

**今天很多岗位、很多讨论，已经被“Prompt Engineer”这个名字误导了。**

两年前这个词还有一定合理性，因为当时大量工作确实集中在：

- 怎么写提示词
- 怎么调输出风格
- 怎么让模型听话一点

但现在 Agent 已经变了：

- 它不只是回答问题
- 它开始执行任务
- 它开始调用工具
- 它开始影响真实世界状态

这时候，“会写 prompt”已经只是入场券。

真正的竞争力会慢慢转向：

- 架构能力
- 工程能力
- 可靠性设计
- 安全治理
- 工作流产品化能力

所以我更认同这个判断：

> **未来真正值钱的，不是 Prompt Engineer，而是 Agent Engineer。**

## 四、我自己的判断：AI Agent 正在把“写提示词”升级成“搭系统”

如果你最近也在看各种 AI Agent 演示，你会发现一个非常典型的错觉：

- 看起来像是模型越来越聪明
- 但真正让系统可用的，往往不是模型本身，而是背后的工程框架

模型当然重要，但模型越来越像“中枢推理层”；真正决定体验上限的，是外围那一整圈：

- 检索
- 工具契约
- 状态管理
- 重试与回滚
- 安全隔离
- 评估与监控
- 用户接管机制

这也是为什么很多人刚开始学 Agent 时会有点错位：

- 以为自己在学 AI
- 学着学着却发现自己绕回了软件工程、系统设计和产品思维

说白了，**Agent 不是把工程替代掉了，而是把工程要求抬高了。**

---

## 五、如果你现在想开始做 Agent，最该先补什么？

我会给一个非常不“花活”的建议：

### 第一优先级：系统设计 + 工具契约
先别急着追逐最复杂的多 Agent 架构，先把：

- 一个 Agent
- 几个稳定工具
- 清晰输入输出
- 明确状态流转

做扎实。

### 第二优先级：检索与可靠性
如果你的 Agent 要读文档、查资料、依赖第三方接口，那这两块会决定它到底是“偶尔聪明”，还是“长期能用”。

### 第三优先级：安全与评估
一旦 Agent 开始碰真实数据和真实动作，就必须把这两块前置，而不是事后补。

## 结语

这支视频最有价值的地方，不是让人学会了 7 个术语，而是它把一个经常被包装得很玄的事情，说回了常识：

> **做能落地的 AI Agent，本质上是在做工程系统，不是在写魔法咒语。**

Prompt 很重要，但它只是开始。

真正能把 Agent 做进生产环境的人，最终拼的是：

- 架构
- 契约
- 检索
- 可靠性
- 评估
- 安全
- 产品化思维

如果你把这件事想明白了，很多焦虑反而会消失。

因为你会发现，所谓“Agent 工程师”，并不是一个突然冒出来的神秘新职业，而是把很多老本事重新组合后，长出来的一种新能力。

而这件事，恰恰最值得认真学。
